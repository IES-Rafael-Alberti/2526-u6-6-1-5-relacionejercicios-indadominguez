### S — SRP (Single Responsibility Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.srp`

Tarea:

- Identifica una clase/función que hace “demasiadas cosas”.
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/50f4bc9c3376f87b0cf7beb2208dfd423f4c3483/src/main/kotlin/es/ies/ejercicios/u6/ej65/srp/SrpV0.kt#L16-L40
  La clase InformeAppServiceV0 contiene toda la responsabilidad del proyecto y lo vamos a dividir en pequeñas partes, cada una con su propia lógica de trabajo.
- Separa responsabilidades en tipos más pequeños (por ejemplo: normalización, formateo, persistencia, logging).
- Mantén un `main`/demo que pruebe el flujo.
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/0bf3d2fd18c95c6161c8153e31823527bdea45e3/src/main/kotlin/es/ies/ejercicios/u6/ej65/srp/SrpV0.kt#L9-L97
Pista:

- Si una clase cambia por más de un motivo (cambiar el formato del informe, cambiar la regla de normalización, cambiar el logging, cambiar cómo se obtienen los datos), probablemente está violando SRP.
- En v0 fíjate en: preparación de datos + registro/normalización + generación de informe + salida por consola, todo junto. Intenta extraer al menos 2 componentes con nombres claros.

Preguntas:

1) ¿Qué responsabilidades tenía la versión v0?  

RESPUESTA:

*La versión v0 tenía múltiples responsabilidades en una sola clase (InformeAppServiceV0):*

- Preparación de datos (creaba la lista de objetos).

- Registro y normalización de personas.

- Generación del informe Markdown.

- Logging y salida por consola.

- Coordinación del flujo completo.

*Esto implica múltiples motivos de cambio:*

- Si cambia el formato del informe.

- Si cambia la forma de registrar personas.

- Si cambia el sistema de logging.

- Si cambia la fuente de datos.

2) ¿Qué clases/componentes has creado y qué responsabilidad tiene cada uno?

RESPUESTA:

*He creado los siguientes componentes:*

- DataProvider
  - Se encarga únicamente de proporcionar los datos.

- PersonaRegistryService
  - Gestiona el registro y búsqueda de personas.

- InformeService
  - Se encarga exclusivamente de generar el informe.

- ConsoleLogger
  - Gestiona la salida por consola (logging).

- InformeAppService
  - Actúa como orquestador del flujo, coordinando los componentes sin implementar lógica interna.

3) ¿Cómo mejora el mantenimiento/testeo con el refactor?

RESPUESTA:

*El refactor mejora el mantenimiento y el testeo porque:*

- Cada clase tiene una única responsabilidad, lo que reduce el acoplamiento.

- Se pueden hacer tests unitarios independientes:

  - Test del registro sin generar informes.

  - Test del informe sin depender del registro.

  - Test del logger por separado.

- El sistema es más extensible:

  - Se puede cambiar InformeMarkdown por otro tipo de informe sin tocar el resto.

  - Se puede cambiar el logger por otro sistema (archivo, base de datos, etc.).

- El código es más limpio, legible y modular.


### O — OCP (Open/Closed Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.ocp`

Tarea:

- El generador v0 usa un `when` por “formato”.
- Refactoriza para que añadir un nuevo formato (p. ej. “HTML” o “JSON”) no requiera modificar el generador central.
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/11ab1858c2fd92f07a672f20fe56334eef6577dc/src/main/kotlin/es/ies/ejercicios/u6/ej65/ocp/OcpV0.kt#L1-L79

Pista:

- Es normal que siga existiendo un punto donde “decides qué implementación crear” (CSV/Markdown/etc.).
- La idea de OCP es que esa decisión no obligue a modificar la **lógica principal** (el generador), sino que se haga fuera (por ejemplo en `main`, una factoría/registry o inyección de dependencias).

Preguntas:

1) ¿Qué parte violaba OCP y por qué?  

RESPUESTA:

La clase GeneradorInformeV0 violaba OCP porque utilizaba un when sobre el enum FormatoInformeV0.

Cada vez que se quería añadir un nuevo formato (por ejemplo HTML o JSON), era necesario:

- Modificar el enum

- Añadir un nuevo caso en el when

- Añadir un nuevo método en la clase

Es decir, la clase no estaba cerrada a modificación, ya que cada nueva extensión obligaba a cambiar código existente.


2) ¿Qué técnica has usado (polimorfismo/estrategia/jerarquía) para cumplir OCP?

RESPUESTA:

*He utilizado:*

- Polimorfismo

- Patrón Strategy

- Una jerarquía basada en una interfaz (FormatoInforme)

Ahora el generador depende de una abstracción y no necesita saber qué formato concreto se está utilizando.


3) ¿Cómo añadirías un nuevo formato tras tu refactor?

RESPUESTA:

*Solo habría que:*

- Crear una nueva clase que implemente FormatoInforme.

- Implementar el método generar.


### L — LSP (Liskov Substitution Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.lsp`

Tarea:

- Hay una jerarquía donde una subclase rompe expectativas del tipo padre (comportamiento inesperado).
- Refactoriza para que cualquier subclase pueda sustituir al padre sin “sorpresas” (sin lanzar excepciones inesperadas, sin romper invariantes).
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/85e23e5828a10bdc8556e36d0c966d07e2725173/src/main/kotlin/es/ies/ejercicios/u6/ej65/lsp/LspV0.kt#L1-L111
Pista:

- Si una “subclase” no puede cumplir el contrato del padre (por ejemplo, “solo lectura” no puede `guardar`), normalmente no debe heredar de ese tipo.
- Una solución habitual es separar interfaces por capacidades (lectura vs escritura) o usar composición para exponer solo lo que realmente se puede hacer.

Preguntas:

1) ¿Qué comportamiento concreto violaba LSP?  

RESPUESTA:

La clase RepositorioSoloLecturaV0 heredaba de RepositorioPersonasV0, pero sobrescribía el método guardar lanzando una excepción UnsupportedOperationException.

*Esto viola LSP porque:*

- El contrato del padre dice que un repositorio permite guardar y buscar.

- El cliente que recibe un RepositorioPersonasV0 espera poder llamar a guardar.

- Al sustituirlo por RepositorioSoloLecturaV0, el comportamiento cambia inesperadamente y lanza una excepción.

2) ¿Qué cambio has hecho para asegurar sustitución válida?

RESPUESTA:

*Se ha eliminado la herencia incorrecta y se han separado las capacidades en interfaces más pequeñas:*

- RepositorioLecturaPersonas

- RepositorioEscrituraPersonas

- RepositorioPersonas (lectura + escritura)

*Ahora:*

- El repositorio completo implementa ambas capacidades.

- El repositorio solo lectura implementa únicamente lectura.

- Ninguna clase implementa métodos que no pueda cumplir.

3) ¿Cómo lo demostrarías con una prueba/ejemplo?

RESPUESTA:

*Se demuestra en el main:*

- RepositorioMemoria puede sustituir a RepositorioPersonas sin problemas.

- RepositorioSoloLectura puede sustituir a RepositorioLecturaPersonas sin problemas.

- Intentar pasar RepositorioSoloLectura a un cliente que requiere escritura no compila.


### I — ISP (Interface Segregation Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.isp`

Tarea:

- La interfaz v0 es “gorda” (métodos no usados por algunos clientes).
- Divídela en interfaces pequeñas y adapta los clientes para depender solo de lo que usan.
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/1c8cec5f53acd97fe8113f5704f623717754fd8d/src/main/kotlin/es/ies/ejercicios/u6/ej65/isp/IspV0.kt#L5-L145

Pista:

- Empieza por el cliente más pequeño: si un cliente solo necesita `buscar(...)`, no debería depender de `exportarCsv()` ni `borrarTodo()`.
- Segrega por capacidades (por ejemplo: lectura, escritura, exportación, borrado) y haz que cada cliente dependa solo de lo que usa.
- Señal típica de violación: ganas de dejar métodos vacíos, lanzar `UnsupportedOperationException` o meter `TODO(...)` al implementar.

Nota (relación con LSP):

- ISP y LSP suelen aparecer juntas: una interfaz “gorda” puede forzar a ciertos tipos a prometer operaciones que realmente no pueden cumplir, lo que termina rompiendo la sustitución (LSP).
- Segregar por capacidades (por ejemplo, lectura vs escritura) ayuda a que los tipos implementen solo contratos que pueden respetar, mejorando el cumplimiento de LSP.

Preguntas:

1) ¿Qué métodos eran irrelevantes para algún cliente?  

RESPUESTA:

*En la versión v0 existía una interfaz con múltiples métodos:*

- Guardar

- Buscar

- ExportarCsv

- BorrarTodo

*El problema era que:*

- Un cliente que solo necesitaba buscar no necesitaba guardar, exportarCsv ni borrarTodo.

- Un cliente que solo daba de alta personas no necesitaba exportar ni borrar.

- Algunos tipos podían verse obligados a implementar métodos que no utilizaban, dejando implementaciones vacías o lanzando UnsupportedOperationException.


2) ¿Cómo has segregado la interfaz (nombres y responsabilidades)?

RESPUESTA:

*Se han dividido las responsabilidades en interfaces pequeñas y específicas:*
 
- RepositorioLectura → solo buscar

- RepositorioEscritura → solo guardar

- RepositorioBorrado → solo borrarTodo

- RepositorioExportacion → solo exportarCsv

Cada cliente ahora depende únicamente de la capacidad que necesita.

3) ¿Qué mejora aporta a largo plazo?

RESPUESTA:

*La segregación aporta:*

- Menor acoplamiento
  - Los clientes dependen solo de lo necesario.

- Mayor claridad en responsabilidades
  - Las capacidades están bien definidas.

- Mejor cumplimiento de LSP
  - Ninguna clase se ve obligada a implementar métodos que no puede cumplir.

- Mejor mantenibilidad
  - Cambiar la exportación no afecta a clientes de lectura.

- Mayor extensibilidad
  - Se pueden crear implementaciones parciales (por ejemplo, un repositorio solo lectura real) sin romper contratos.

### D — DIP (Dependency Inversion Principle)

**Código v0:** `es.ies.ejercicios.u6.ej65.dip`

Tarea:

- Un módulo de alto nivel depende directamente de una implementación concreta.
- Refactoriza para que dependa de **abstracciones** e inyecte implementaciones (constructor recomendado).
  https://github.com/IES-Rafael-Alberti/2526-u6-6-1-5-relacionejercicios-indadominguez/blob/cc07feecc13adb73eb6021137e5fe51fc6300306/src/main/kotlin/es/ies/ejercicios/u6/ej65/dip/DipV0.kt#L1-L83

Pista:

- “Alto nivel” = orquesta el caso de uso; “bajo nivel” = detalles concretos (CSV/Markdown, consola/archivo, etc.).
- Crea una abstracción (interfaz) que represente lo que necesitas (p. ej. `GeneradorInforme`), y pasa la implementación por **constructor**.
- Si para cambiar de CSV a Markdown tienes que editar la clase de alto nivel, aún no estás cumpliendo DIP.

Preguntas:

1) ¿Cuál era el detalle concreto del que dependía el módulo de alto nivel?  

RESPUESTA:

*En la versión v0, el módulo de alto nivel (por ejemplo InformeAppService) dependía directamente de una implementación concreta como:*

- GeneradorInformeCsv

- GeneradorInformeMarkdown

2) ¿Qué abstracción has creado/empleado?

RESPUESTA:

*Se ha creado la interfaz GeneradorInforme*

3) ¿Cómo intercambiarías implementaciones sin tocar el módulo de alto nivel?

RESPUESTA:

*Solo cambiando la implementación en el main (composición):*


El InformeAppService no necesita modificarse en absoluto.

El cambio ocurre fuera del módulo de alto nivel, normalmente en:

- Main

- Una factoría

- Un contenedor de dependencias

- Configuración externa
